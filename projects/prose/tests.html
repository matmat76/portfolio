Plan de test

La phase de test du projet ProSE, en particulier pour le composant Catalog, est planifiée pour la période du 1er mai au 20 juin 2025. Cette phase est critique pour assurer la qualité et la fiabilité du système avant la livraison finale prévue le 12 juin.

1er - 20 mai 2025

Développement des tests unitaires

Conception et implémentation des tests unitaires pour valider le comportement de chaque fonction du module Catalog.

15 - 31 mai 2025

Tests d'intégration

Vérification des interactions entre le Catalog et les autres composants du système.

1er - 10 juin 2025

Tests de performance

Évaluation des temps de réponse et de la consommation de ressources sous différentes charges.

12 juin 2025

Livraison finale

Remise du système complet et de la documentation associée.

Tests unitaires

Les tests unitaires constituent la base de notre stratégie de test. Ils permettent de valider le comportement de chaque fonction et de chaque composant de manière isolée.

Framework de test

Le framework Check est utilisé pour implémenter les tests unitaires en C. Ce framework fournit des macros d'assertion et une structure organisée pour l'exécution des tests.

Exemple de test unitaire pour la fonction catalog_init() 
#include <check.h>
#include "../src/catalog.h"

START_TEST(test_catalog_init) {
    catalogue_t catalog;
    int catalog_id = 42;
    
    // Initialiser le catalogue
    catalog_init(&catalog, catalog_id);
    
    // Vérifier que l'initialisation définit bien les valeurs par défaut
    ck_assert_int_eq(catalog.id, catalog_id);
    ck_assert_int_eq(catalog.state, CATALOG_LOADING);
    ck_assert_int_eq(catalog.nb_videos, 0);
    ck_assert_int_eq(catalog.nb_projets, 0);
    ck_assert_int_eq(catalog.nb_collaborateurs, 0);
    ck_assert_ptr_null(catalog.videos);
    ck_assert_ptr_null(catalog.projets);
    ck_assert_ptr_null(catalog.collaborateurs);
}
END_TEST

Principaux cas de test

Voici les principales catégories de tests unitaires qui sont en cours de développement :

  • Tests d'initialisation : Valider que l'initialisation configure correctement les structures
  • Tests de la machine à états : Vérifier les transitions d'état et le comportement associé
  • Tests de chargement de données : Valider le chargement des différents types de contenu
  • Tests de sélection d'items : Vérifier l'algorithme de sélection des prochains items à afficher
  • Tests de notification : Valider le système de callback pour les notifications
  • Tests de gestion de mémoire : Vérifier l'allocation et la libération des ressources

Tests d'intégration planifiés

Les tests d'intégration sont essentiels pour valider le fonctionnement harmonieux des différents composants assemblés. Ces tests seront réalisés à l'aide de Robot Framework, un outil d'automatisation de tests qui permet de créer des scénarios de test complets.

Robot Framework

L'utilisation de Robot Framework pour les tests d'intégration permettra d'automatiser les scénarios complexes et de valider le comportement du système de bout en bout. Les scénarios suivants sont planifiés :

Scénario 1: Configuration et affichage d'un nouveau catalogue

Ce scénario testera le parcours complet depuis la configuration d'un nouveau catalogue dans FortilApp jusqu'à l'affichage des contenus sur l'écran.

État: Planifié

Scénario 2: Changement dynamique de catalogue

Ce scénario testera la capacité du système à basculer d'un catalogue à un autre sans redémarrage ni interruption visible.

État: Planifié

Scénario 3: Récupération après erreur réseau

Ce scénario vérifiera le comportement du système en cas de perte de connexion avec le serveur et sa capacité à reprendre l'affichage une fois la connexion rétablie.

État: Planifié

Tests de performance avec JMeter

Pour évaluer les performances de la communication HTTP avec le serveur, JMeter sera utilisé pour simuler des charges variées et mesurer les temps de réponse.

Objectifs des tests de performance

  • Évaluer le temps de réponse sous différentes charges (1, 10, 50 requêtes simultanées)
  • Identifier les goulots d'étranglement potentiels
  • Mesurer la stabilité du système sur de longues périodes
  • Valider la robustesse en cas de pics de charge

Plan de test JMeter

Le plan de test JMeter comportera plusieurs groupes d'utilisateurs virtuels simulant différentes actions :

  • Utilisateurs récupérant des catalogues
  • Utilisateurs téléchargeant des vidéos
  • Utilisateurs récupérant des informations de collaborateurs
  • Utilisateurs consultant des fiches projet

Ces tests permettront d'identifier les performances réelles du système et de proposer des optimisations si nécessaire.

Tests de montée en charge

En plus des tests fonctionnels et d'intégration, des tests de montée en charge sont prévus pour valider le comportement du système dans des conditions de stress :

Test de charge constante

Le système sera soumis à une charge constante sur une longue période (plusieurs heures) pour vérifier sa stabilité et détecter d'éventuelles fuites de mémoire.

Test de pics de charge

Des pics de charge soudains seront simulés pour valider la capacité du système à gérer des variations importantes d'utilisation sans dégradation significative des performances.

Test de récupération

Après une défaillance simulée, la capacité du système à revenir à un état stable et à récupérer les données sera évaluée.

Critères de réussite

Pour que la phase de test soit considérée comme réussie, les critères suivants devront être satisfaits :

  • Couverture de code d'au moins 85% pour le module Catalog
  • Temps de réponse inférieur à 200ms pour les opérations courantes
  • Stabilité du système sur des périodes prolongées (> 24h)
  • Aucune fuite de mémoire détectée
  • Récupération automatique après perte de connexion
  • Réussite de tous les scénarios de test d'intégration
