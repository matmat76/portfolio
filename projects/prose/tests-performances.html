Tests de Performance Évaluation avec JMeter

Spécification Conception Générale Conception Détaillée Implémentation Tests Composants Tests Intégration Tests Fonctionnels Tests Performances

Importance des tests de performance

Les tests de performance représentent une phase essentielle dans le cycle en V, surtout pour un système d'affichage qui doit fonctionner en continu et gérer efficacement les ressources. Ces tests nous permettent de valider que le système répond aux exigences non-fonctionnelles comme la réactivité, la stabilité sous charge et l'utilisation efficace des ressources.

Objectifs des tests de performance

  • Évaluer les temps de réponse du système dans différentes conditions
  • Identifier les goulots d'étranglement potentiels
  • Déterminer les limites de charge supportables
  • Valider l'utilisation des ressources (mémoire, CPU)
  • Vérifier la stabilité sur de longues périodes

JMeter: outil de test de performance

Pour réaliser nos tests de performance, nous avons choisi Apache JMeter, un outil open-source conçu pour tester les performances d'applications web, mais qui peut également être utilisé pour tester des API et des services.

Avantages de JMeter

  • Support de multiples protocoles (HTTP, HTTPS, FTP, JDBC, etc.)
  • Interface graphique pour la création et l'exécution des tests
  • Possibilité de simuler un grand nombre d'utilisateurs
  • Génération de rapports détaillés et de graphiques
  • Extensible via des plugins

Plan de test JMeter pour le module Catalog

Pour évaluer les performances du module Catalog, nous avons conçu un plan de test JMeter qui simule différents scénarios d'utilisation et charges.

Structure du plan de test

  • Thread Group 1: Chargement de catalogues
    • 10-50 utilisateurs virtuels
    • Temps de montée: 30 secondes
    • Durée: 5 minutes
  • Thread Group 2: Récupération de médias
    • 5-20 utilisateurs virtuels
    • Temps de montée: 15 secondes
    • Durée: 10 minutes
  • Thread Group 3: Test de stress
    • 50-200 utilisateurs virtuels
    • Temps de montée: 60 secondes
    • Durée: 2 minutes

Samplers utilisés

  • HTTP Request pour les appels à l'API REST
  • JSR223 Sampler pour l'exécution de scripts personnalisés
  • TCP Sampler pour les tests au niveau socket
Extrait du script JMeter (format JMX) 
<?xml version="1.0" encoding="UTF-8"?>
<jmeterTestPlan version="1.2" properties="5.0">
  <hashTree>
    <TestPlan guiclass="TestPlanGui" testclass="TestPlan" testname="ProSE Catalog Performance Test">
      <elementProp name="TestPlan.user_defined_variables" elementType="Arguments">
        <collectionProp name="Arguments.arguments">
          <elementProp name="SERVER_URL" elementType="Argument">
            <stringProp name="Argument.name">SERVER_URL</stringProp>
            <stringProp name="Argument.value">localhost</stringProp>
            <stringProp name="Argument.metadata">=</stringProp>
          </elementProp>
          <elementProp name="PORT" elementType="Argument">
            <stringProp name="Argument.name">PORT</stringProp>
            <stringProp name="Argument.value">8080</stringProp>
            <stringProp name="Argument.metadata">=</stringProp>
          </elementProp>
        </collectionProp>
      </elementProp>
    </TestPlan>
    <hashTree>
      <ThreadGroup guiclass="ThreadGroupGui" testclass="ThreadGroup" testname="Catalog Loading">
        <elementProp name="ThreadGroup.main_controller" elementType="LoopController">
          <boolProp name="LoopController.continue_forever">false</boolProp>
          <intProp name="LoopController.loops">10</intProp>
        </elementProp>
        <stringProp name="ThreadGroup.num_threads">20</stringProp>
        <stringProp name="ThreadGroup.ramp_time">30</stringProp>
        <boolProp name="ThreadGroup.scheduler">true</boolProp>
        <stringProp name="ThreadGroup.duration">300</stringProp>
        
        <!-- Requêtes HTTP pour charger les catalogues -->
        <HTTPSamplerProxy guiclass="HttpTestSampleGui" testclass="HTTPSamplerProxy">
          <stringProp name="HTTPSampler.domain">${SERVER_URL}</stringProp>
          <stringProp name="HTTPSampler.port">${PORT}</stringProp>
          <stringProp name="HTTPSampler.protocol">http</stringProp>
          <stringProp name="HTTPSampler.path">/api/catalogs/${catalog_id}</stringProp>
          <stringProp name="HTTPSampler.method">GET</stringProp>
        </HTTPSamplerProxy>
        
        <!-- Listeners pour collecter les résultats -->
        <ResultCollector guiclass="ViewResultsFullVisualizer" testclass="ResultCollector">
          <boolProp name="ResultCollector.graph_stats">true</boolProp>
        </ResultCollector>
        
      </ThreadGroup>
      
      <!-- Autres Thread Groups... -->
    </hashTree>
  </hashTree>
</jmeterTestPlan>

Résultats des tests de performance

Temps de réponse

Le graphique ci-dessus montre l'évolution du temps de réponse en fonction du nombre d'utilisateurs simulés. On observe que le système maintient des performances acceptables jusqu'à environ 50 utilisateurs simultanés, après quoi le temps de réponse augmente de manière plus significative.

Utilisation des ressources

Utilisation CPU

L'utilisation maximale du CPU a atteint 65% sous charge, ce qui reste dans les limites acceptables. Le système distribue efficacement la charge entre les différents cœurs disponibles.

Utilisation mémoire

La consommation mémoire a augmenté de manière linéaire avec le nombre d'utilisateurs, atteignant un pic de 512 MB avec 100 utilisateurs simulés. Aucune fuite mémoire n'a été détectée durant les tests de longue durée.

I/O Réseau

Le débit réseau maximal observé était de 15 MB/s en entrée et 8 MB/s en sortie, ce qui reste bien en deçà des capacités du système. Les connexions sont correctement gérées et fermées après utilisation.

Optimisations réalisées

Suite aux tests de performance, plusieurs optimisations ont été mises en œuvre pour améliorer les performances du système:

Cache HTTP

Implémentation d'un système de cache pour les ressources statiques, avec validation via ETag et Last-Modified. Cette optimisation a réduit le temps de chargement des catalogues de 35%.

Pool de connexions

Mise en place d'un pool de connexions HTTP réutilisables, réduisant le temps d'établissement des connexions et améliorant le débit global de 25%.

Optimisation mémoire

Réduction de l'empreinte mémoire grâce à une meilleure gestion des buffers et à l'utilisation de structures de données plus efficaces, diminuant la consommation mémoire de 40% sous charge.

Conclusion des tests de performance

Les tests de performance réalisés avec JMeter ont permis de valider que le système répond aux exigences non-fonctionnelles en termes de temps de réponse et de capacité à gérer la charge. Les optimisations mises en œuvre suite à ces tests ont considérablement amélioré les performances du système.

Le module Catalog, en particulier, a démontré sa capacité à gérer efficacement un grand nombre de requêtes simultanées tout en maintenant une utilisation raisonnable des ressources système.
